# Opening Entry Design

このドキュメントは、期首仕訳をどのレイヤーで扱うかと、その登録ルールを整理したものです。

## 目的

- 期首仕訳をセットアップ画面の副作用ではなく、明示的な業務 API として扱う
- 通常仕訳と共通化できる保存処理は再利用しつつ、期首仕訳固有の制約は専用の入口に閉じ込める
- 将来の差替・取消・編集機能追加に備えて、責務の境界を先に固定する

## 入口

期首仕訳のアプリケーション上の入口は `App\Services\OpeningEntryRegistrar` とする。

- `FiscalYear::registerOpeningEntry()` は薄い委譲メソッドとして残す
- 初期セットアップは `OpeningEntryRegistrar` の利用者の 1 つとして扱う
- `TransactionRegistrar` は保存基盤として再利用する

したがって、期首仕訳を作る実装は `Transaction::create()` や `JournalEntry::create()` を直接呼ばず、`OpeningEntryRegistrar` を経由する。

## Registrar の責務

`OpeningEntryRegistrar` は期首仕訳専用のユースケース層として扱う。

- 空入力時は何も作らず `null` を返す
- 同一会計年度に既存の期首仕訳があるか確認する
- 取引日を会計年度の `start_date` に固定する
- `description` を `期首残高設定` に固定する
- `is_opening_entry = true` を必ず付与する
- 借方に使える勘定科目を制限する
- 必要な補助科目を自動作成または再利用する
- 貸方を `元入金` に集約する
- 永続化自体は `TransactionRegistrar` に委譲する

## 現在の登録ルール

現時点の期首仕訳は、個人事業の初期残高登録を前提に次のルールで扱う。

- 1 会計年度につき期首仕訳は 1 伝票まで
- 借方に使える勘定科目は `現金` `定期預金` `その他の預金` `車両運搬具` `棚卸資産` のみ
- 貸方は常に `元入金`
- 各借方明細は `sub_account_name` と正の `amount` を必須とする

## レイヤー分離

責務分離は次のようにする。

- `OpeningEntryRegistrar`
  - 期首仕訳固有の業務ルール
- `TransactionRegistrar`
  - 共通の取引バリデーションと保存
- `SetupWizard` / `GeneralBusinessInitializer`
  - 初期入力の収集と API 呼び出し

この分離により、セットアップ以外の UI や API を追加しても、期首仕訳の登録ルールを再実装せずに済む。

## 将来拡張の論点

今後、次の論点は専用 API のまま拡張する想定とする。

- 期首仕訳の差替を許可するか
- 既存期首仕訳の取消後に再登録するか
- `元入金` 固定を維持するか
- 負債や未払金などを含むより一般的な開始残高へ広げるか
- 会計年度締め後の変更を禁止するか

現時点では、まず「明示的な登録 API を持つこと」と「同一年度で重複登録できないこと」を優先する。
